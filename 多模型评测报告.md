# HarmonyOS ArkTS DeepSeek 聊天应用多模型评测报告

> 评测日期：2026-06-28
> 评测对象：GPT-5.5、Claude Opus 4.8、Gemini 3.5 Flash、Composer 2.5、DeepSeek V4 Pro、DeepSeek V4 Flash、GLM 5.2、Qwen 3.7 Max、豆包 2.1 Pro、MiniMax M3、Kimi K2.7 Code、MiMo 2.5 Pro、ERNIE-5.0-Thinking-Preview
> 题目：在给定 HarmonyOS ArkTS 示例工程中完成可由 DevEco 编译的 DeepSeek AI 聊天 APP
> 扣分口径：累计开发过程中每一次构建输出的 ArkTS 编译错误；修复后新增或复现的错误同样再次扣分
> 报告性质：代码审查、Git 过程审计、编译复核与已有真机/运行记录的综合评测

---

## 一、结论摘要

本次测评不把“代码量多”“页面看上去齐全”或“最后一次可以编译”当作完成任务的替代品。题目要求的是一个可直接在 DevEco 编译、能承担真实对话流程的 ArkTS 应用；因此必须同时考察首次交付可编译性、HTTP/SSE 的实现方式、思考内容的显示与回传、多轮历史、工具调用协议、UI 的实时状态更新，以及交付过程是否表现出对 ArkTS 约束的掌握。

综合得分第一名仍是 **GPT-5.5，91 分**。它是全部十三个评分模型中唯一首次提交即编译通过的方案，最终仓库也经本次独立复编译成功。其工程分层、SSE 增量解析、页面反馈、历史保存和设置交互都较完整，界面质量也得到运行记录的正面评价。它并非没有可改进处：工具回合中没有把已获得的 `reasoning_content` 重放为 API 助手消息字段，若服务端在“开启思考 + 工具调用”组合下要求该字段回传，存在同类协议风险；另外没有专门测试用例，且有若干 ArkTS 警告。但这些属于上线前的明确修补项，不改变其明显领先的事实。

第二名为 **Claude Opus 4.8，79 分**。它首交只有 5 个 ArkTS 编译错误，编译稳定性仍处在前列；最终使用 `requestInStream`，思考和正文都有增量回调，工具调用也在多次修复后跑通。缺点同样明确：工具问题前两轮修复明显在猜根因，直到服务端返回“assistant tool_calls 后必须跟 tool message”的协议错误后才真正补上消息顺序；最终模型层保存了 reasoning，但 `ApiMessage` 没有 `reasoning_content` 字段，仍保留思考模式工具回合的潜在协议风险。

第三名为 **Gemini 3.5 Flash，66 分**。它首交有 14 个 ArkTS 编译错误，最终复编通过；更重要的是最终流程能跑通，且代码采用 `requestInStream`，消息模型显式保留 `reasoning_content`、`tool_calls`、`tool_call_id`，比许多旧模型更接近思考模式工具回合的正确协议。扣分点是第一次修复后出现“流式阶段正常显示、结束后新消息消失、思考只剩最近一次工具后的内容”的 UI 状态错误，说明页面状态与持久化闭环曾经断裂；最终实现也只执行第一个工具调用，SSE/UTF-8 分片处理仍有边界风险。

第四名为 **Composer 2.5，64 分**。它首交只有 3 个 ArkTS 编译错误，最终构建通过，设置、历史、思考开关、high/max、时间工具和工具后最终回答都能形成业务闭环；代码层还在 `ApiChatMessage` 与 `ChatManager` 中保留并回传 `reasoning_content`，这是许多高分方案都遗漏的协议细节。它不能更高的原因也很明确：最终仍使用普通 `httpRequest.request()`，当 API 23 不逐 token 触发 `dataReceive` 时，解析完整 `resp.result` 后交给 `StreamDisplayQueue` 以 16ms 定时器做“打字机”展示；这改善了观感，但不是题目要求的客户端真流式。因此它凭极低编译扣分排到 DeepSeek V4 Pro 之前，但不能越过真正使用流式链路的 Gemini。
第五名为 **DeepSeek V4 Pro，62 分**。它最终业务流程跑通，最终源码也有 `requestInStream`、分片 SSE、思考展示、工具调用累积与工具结果回填，完成度高于多数旧模型；但首交 19 个 ArkTS 编译错误，一开始还把 INTERNET 权限问题留给用户“手动添加”，之后又经历无响应、权限失败、工具后无响应三轮修复，说明首次交付和自检质量明显不足。按全程错误累计扣分后，它仍能排到 GLM 前面，但不是高稳定性交付。

第六名为 **GLM 5.2，58 分**。其最终版本编译通过，后续 Git 变动仅为两轮小修，修复收敛性相对较好，且使用了 `requestInStream`、SSE 解析器、设置和历史页面。最大问题是开发过程中累计有 14 个 ArkTS 编译错误，暴露了对严格类型、`Preferences` 返回类型、组件 API 和 `requestInStream` 回调签名的不熟悉；功能实现的工具回合也没有保留/回传 reasoning 内容，因而不能将“已准备好工具调用”给到满分。

第七名为 **Qwen 3.7 Max，51 分**。界面和功能面铺得较全，设置、历史和工具的轮廓均具备，最终也可编译；但第一次有 11 个编译错误，且运行记录已经证实它并未让用户获得真实流式输出，工具调用仍报错。根因可从源码复核：`DeepSeekService.ets` 采用 `httpRequest.request()` 接收完整响应后再解析 SSE，而不是用 `requestInStream`；这不是“动画不够顺滑”，而是把题目“流式输出”的核心实现路径换成了批量返回后的伪流式处理。

第八名为 **豆包 2.1 Pro，39 分**。它最终有真正的流式请求和较完整的数据模型、设置页、历史页、工具定义，代码组织在多次修复后的形态也不差；然而首次提交有 24 个编译错误，随后为修复编译与“模型无回复”修改了 715 行，首次交付质量显著不足。运行记录还确认最终界面存在文字与容器错位、文字越出背景或屏幕的重大可用性问题，工具回合也未闭合，因此不能仅因最终构建成功而给出高分。

第九名为 **DeepSeek V4 Flash，25 分**。它最终也跑通了业务流程，并且源码使用 `requestInStream` 处理流式响应，设置、历史、思考开关、high/max、时间工具都具备，不属于伪流式路线。但它的首次交付质量非常差：首轮 39 个 ArkTS 编译错误，随后两轮修复又分别产生 8 个和 3 个错误，累计 50 分硬扣；编译通过后还先出现一长串 `nullnullnull` 污染回复、工具回合因 assistant/tool 消息顺序不满足协议而 400。最终能收敛值得肯定，但大面积修编译和协议后修补说明首交自检不足，故排在豆包之后、MiniMax/Kimi之前。

第十名为 **MiniMax M3，23 分**。它最终能构建，也实现了请求流、工具循环和存储结构，但累计 24 个编译错误后发生了大规模重写，整个交付过程多次把“完成(stop,1轮)”误判为用户已经看到了回复。实测显示模型内容不显示、需重载才可见、工具名反复为空且在思考模式后得到服务端明确的 `reasoning_content ... must be passed back` 400 错误。其最终源码的 `messageToJson()` 也印证：助手工具消息没有携带 reasoning 字段。

第十一名为 **Kimi K2.7 Code，22 分**。它在旧模型中是源码层面对 DeepSeek 思考字段理解最接近正确的一组：`buildApiMessages()` 会将 `reasoningContent` 赋给 `reasoning_content`，并且工具调用有轮次上限、工具结果回填和 `requestInStream`。但这种优点被累计 37 个编译错误、连续七次修复、发送入口失效、会话为空、中文乱码和“收到后只在重新加载时显示”的实测问题大幅抵消。尤其是服务层虽累积增量，却没有在收到增量时调用 `onContentDelta`/`onReasoningDelta` 更新 UI；结果符合运行记录中的“并非实时输出”。

第十二名为 **MiMo 2.5 Pro，4 分**。它有 README、多个独立页面和较多表面功能，但累计 41 个编译错误；此后七次修补又修改 765 行，仍以 `http.request()` 读取完整 SSE 文本，无法满足实时流式要求。运行记录还指出思考不可见、工具结果不能正确回传、后续请求 400；中途将“`http.request()` 无法真正流式”包装成限制，是对现有 `requestInStream` 能力的错误判断。最终能编译只说明语法暂时被修平，不能证明业务完成。

第十三名为 **ERNIE-5.0-Thinking-Preview，-78 分**。它最终 HEAD 可以编译，但运行记录最终停在纯白界面，没有可验收的主流程。更严重的是开发过程中累计 102 个 ArkTS 编译错误，且修复时经常不是定位局部根因，而是推翻/重写文件、再引入新的语法或 ArkUI 结构错误；这正是用户额外指出的 API 消耗量大的原因。它还使用普通 `http.request()` 接收完整响应后解析 SSE，没有真实流式，工具也没有按题目要求形成工具调用闭环，因此成为本组唯一负分样本。

## 二、题意拆解与评测边界

题目要求一个完整 HarmonyOS 手机 APP，必须支持以下能力：显示思考、多轮对话、流式输出；由用户决定是否开启思考并选择 `high` 或 `max`；保存和读取历史；为工具调用预留并实现获取当前手机系统时间；允许用户填写 API Key；不使用现成 OpenAI SDK，而是以 ArkTS 自行发 HTTP 请求；请求模型为 `deepseek-v4-flash` 或 `deepseek-v4-pro`，并以 `stream: true` 请求。

这些文字并不只是若干 UI 控件。它们至少蕴含六条端到端契约：

1. **配置契约。** API Key、模型、思考开关和深度必须从设置页进入运行时配置，保存后重新打开仍可读取；关闭思考时应发送 `thinking: { type: 'disabled' }`，启用时才应带 `reasoning_effort`，并限制取值为 `high`/`max`。
2. **网络契约。** 请求必须发往 `/chat/completions`，包含 Bearer API Key、`stream: true`、消息数组和工具定义；对于连续响应，不能等待全部 body 到齐后才更新用户界面。
3. **SSE 契约。** 同一个 `dataReceive` 分片可能包含半条事件、零到多条事件，UTF-8 字符也可能跨分片。实现要维护缓存，以空行/行边界切割事件，处理 `data:`、`[DONE]`、普通内容、`reasoning_content` 和增量 `tool_calls`。
4. **会话契约。** 每轮用户消息、助手消息、工具调用消息和工具结果必须以正确顺序进入会话；下一轮请求应得到完整而合法的历史；保存和重新加载后显示与请求上下文应一致。
5. **工具契约。** 服务返回工具调用后，客户端应合并按索引切片的 id、函数名、参数，执行本地时间工具，将带 `tool_call_id` 的 `role: tool` 消息放回历史，并再次发起请求获得最终自然语言回答。开启思考时，若服务需要前一助手消息的 `reasoning_content`，同样必须随工具回合一并回传。
6. **产品契约。** 用户要看得见正在思考、正在生成、工具执行、失败、停止和完成，而不是只有日志或内部计数；文案可读、中文不乱码、控件不越界、返回与历史切换可达。

本报告不要求各方案的视觉设计完全相同，也不因未接入真实密钥而把网络请求失败计为功能失败。评价的是实现是否具备真实运行所需路径，以及 `结果.md` 已记录的带真实交互的观察是否与源码相一致。题目没有要求云端部署、账号体系、加密 Keychain、Markdown 渲染、图片输入或自动化 CI；这些不会成为额外的硬性扣分项。不过，对 API Key 明文持久化、缺少测试、无签名配置等影响上线可靠性的事项仍会在代码质量部分说明。

## 三、证据来源、复核方法与可信度说明

### 3.1 使用的证据

本报告使用四类相互校验的证据，而不是只采信模型自己的完成声明。

* **题目文件 `question.md`**：用于确定必须功能、API 形态、模型名和硬性 `stream: true` 要求。
* **运行记录 `结果.md`、`额外结果.md`、`额外额外结果.md`、`v4flash结果.md` 与 `composer结果.md`**：其中保存了每份首次交付的 ArkTS 编译输出，以及在模拟器/设备上发送、重载、调用工具后的现象。诸如“点击发送没有反应”“重载后才看到”“HTTP 400 的服务端文本”“文字跑到屏幕外”均为优先级很高的黑盒证据。
* **每个子目录中的独立 Git 仓库**：检查从项目初始提交到首次交付、再到最终 HEAD 的提交序列、差异行数和变更范围。这样可以区分小范围纠错与先交不可用代码、后以大重写补救的情况。
* **最终 HEAD 源码与本次编译**：重点阅读 `DeepSeekService`/`DeepSeekClient`、SSE/HTTP 工具、消息模型、存储层和主页面的调用链；随后在 `D:\DevEcoStudio` SDK 下执行 `hvigorw.bat assembleHap --no-daemon` 对全部最终仓库复编译。

复编译所使用的环境变量为 `DEVECO_SDK_HOME=D:\DevEcoStudio\sdk`，Node 路径为 DevEco 自带工具目录。十三个最终 HEAD 的 `assembleHap` 均返回成功，工作区 `git status --porcelain` 也保持干净。构建日志提示项目未配置签名配置而跳过 HAP 签名；这是示例工程常见状态，意味着“构建与打包通过”，不是“已经按生产证书签名发布”。GPT 的构建另有若干未处理异常和已弃用 API 警告，报告没有把警告按题目“每个编译错误扣一分”的规则折算为错误扣分，但在质量判断中保留影响。

### 3.2 不能从本次证据推出的事情

没有为十三个仓库分别使用真实 API Key 再跑一套自动化端到端脚本，也没有将 HAP 安装到同一台实体手机进行逐像素截图对比。因此，对 `结果.md`、`额外结果.md`、`额外额外结果.md`、`v4flash结果.md` 与 `composer结果.md` 中已有的真实交互现象，本报告视为已观察证据；对源码才能发现而尚未再次触发的缺陷，明确标成“源码风险”或“潜在风险”，不伪装成已发生的真机错误。尤其是 GPT 的 reasoning 回传问题属于这种第二类：源码不包含字段，结合 MiniMax 的服务端错误可以判定为需要修复的协议风险，但不与“已实测必现”混写。

同理，首次提交的错误数以 `结果.md` 中标明的“首次提交/首次交付/首次提交代码”编译记录为准，而不是以最终 HEAD 反推。GLM 的首个 Git 提交同时名为“新建项目+首次提交”，MiniMax 的首个提交名为“创建项目+首次提交”；它们均包含题目实现，故均被当作首次交付。后续“修复编译错误”的提交不是首次提交，不能擦掉扣分。

### 3.3 为什么要审 Git，而不只审最终代码

这道题刻意选择 ArkTS，测试的不只是通用 TypeScript 生成能力，而是模型能否在强约束语言、HarmonyOS SDK API 和异步 UI 框架下先形成可构建的设计。一个模型先写出几十个不能编译的对象字面量、`any` 风格数据和错误 API 签名，再经数轮提示反复修补，反映出任务分解、接口查证与自检不充分。反过来，首次就通过也不意味着代码天然正确，仍须做协议和业务检查。

过程审计并不是机械惩罚“修改得多”。复杂功能可以合理迭代；真正的负面信号是修复跨度大、修改触及所有核心层、运行故障从编译迁移到发送/状态/协议、并且模型把错误状态误宣称为完成。例如 MiniMax 在首次交付后总计新增 637 行、删除 516 行，MiMo 后续新增 507 行、删除 258 行；这些数字本身不是罪，但与多次可复现的主流程故障合在一起，说明初始架构没有验证通过。ERNIE 的增删行数不大，却在 14 个修复提交里反复推翻同一两个核心文件，更能暴露定位能力不足。

## 四、评分规则

### 4.1 分值结构

基础分总分 100，分为四个维度；在此之后一次性减去开发过程中累计出现的 ArkTS 编译错误数。公式为：

```text
最终得分 = 功能完成度 + 业务流程质量 + 代码质量 + 工作过程质量
           - 各次构建中 ArkTS Compiler Error 数量之和
```

### 4.2 A：功能完成度（30 分）细则

* 设置与模型选择 6 分：Key 输入、`flash`/`pro` 选择、思考开关、`high/max`、保存读取各有相应分值。
* 真流式与思考展示 8 分：必须使用可逐分片处理的 API；必须将 `reasoning_content` 与常规内容分别处理并在 UI 实时显示。只把完整 SSE 字符串拆开后一次性渲染，至多给协议解析的少量分，不给真流式分。
* 历史与多轮 5 分：用户、助手、工具三类消息的保存、恢复、切换和下一轮上下文必须一致。
* 工具 6 分：工具 schema、参数拼接、本地执行、`tool_call_id`、回填、再次请求和最终回答缺一不可；“能出现工具图标”不等于工具闭环。
* 错误、取消和基本产品完成度 5 分：空 Key、网络异常、取消发送、等待状态、中文可读和基本可操作性。

### 4.3 B：业务流程质量（30 分）细则

本维度强调“用户一路走下来是否能完成任务”。首轮提问后要看到增量思考/回答；连续追问要带上上下文；退出重进要能找回历史；问“现在几点”后要看到工具执行与最终回答；失败要可解释、可再次发送，而不是挂在“完成”或空白状态。已有运行记录中确认的主流程故障按严重度扣分：不能发送、只重载后显示、工具后 400、中文乱码、布局遮挡、假流式、未知工具重复等均比代码风格问题严重。

### 4.4 C：代码质量（25 分）细则

ArkTS 不允许把传统 TypeScript 的宽松做法直接搬来。高分实现应为对象字面量提供显式接口/类，避免错误的 `Object`/`Record<string, Object>` 强转扩散，正确处理 `preferences.ValueType`，并核实 NetworkKit 中 `requestInStream` 的真实签名。HTTP/SSE 实现应有可读的数据模型、分片缓存、UTF-8 解码、`[DONE]` 与错误 body 处理、`destroy()` 释放，工具切片应按 `index` 合并。存储层应捕获 JSON 损坏和 Preferences 异常，UI 层不应依赖重载来刷新状态。

密码型输入框可以避免旁观者直接看到 API Key，但若把 Key 与整个对话 JSON 明文存入 Preferences，仍是生产安全债务。题目只要求用户填写，并没有指定安全存储服务，所以此项作为质量扣点而非致命不合格。十三个方案都没有新增针对 SSE、工具 JSON 或持久化迁移的专项测试；模板自带测试文件不能算作新增覆盖，故该子项普遍偏低。

### 4.5 D：工作过程质量（15 分）细则

本项奖励在写代码前确认 ArkTS/NetworkKit API、先完成一条最小可构建垂直链路、再扩展页面的做法。首次提交是否能编译在本项有正向分，开发过程中每次出现的编译错误还会在硬扣分中累计体现；这不是重复惩罚，而是分别衡量“过程可靠性”和“题目明确规定的数学惩罚”。大量后续修改不自动扣尽分，只有在它显示核心设计返工、修复引入新故障、或没有验证就反复宣称完成时才显著降分。

该版本的优势不止是“能编译”。模型层 `ChatModels.ets` 把角色、模型 ID、深度、会话、工具调用分开声明；服务层 `service/DeepSeekService.ets` 封装请求、取消、SSE 解析和回调；持久化独立在 `ChatStorage.ets`；页面层 `pages/Index.ets` 负责会话列表、设置抽屉和渲染。这种结构使得调用链很容易读：`send()` 追加用户消息，`runRound()` 发起流，内容与思考分别进入 `streamingContent` 和 `streamingReasoning`，遇到工具后追加助手工具消息和 `role: tool` 消息，再递归进入下一轮。

它正确使用了 `http.createHttp()`、`requestInStream()`、`dataReceive`、`dataEnd` 与 `util.TextDecoder`。服务层不假设一个数据块就是一个 SSE 事件，而是用双换行事件边界切出完整 SSE event，再筛选 `data:` 行；这比按一次普通换行立即解析稳健得多。`tool_calls` 则按服务器给出的 `index` 合并 id、函数名和参数；这正是流式工具调用最容易遗漏的地方。`stop()` 会 destroy 当前 request，UI 也会把已收到的部分回答保存，这些都体现出对用户中断操作的考虑。

产品层也完成了题意中最常被忽略的细节：Key 为空时自动打开设置并提示；Key 输入使用 `InputType.Password`；思考块可以展开/收起；新会话、历史会话、当前标题和正在调用时间工具都有可见反馈；模型可在 `deepseek-v4-flash` 与 `deepseek-v4-pro` 间选择，思考关闭时不会带深度选择。运行记录对其评价为“整套业务流程完全正常，界面美观，可作为 APP 落地”，这是本组唯一接近该等级的黑盒结论。

需要修复的第一项是 **工具回合的 reasoning 消息重放**。`ChatMessage` 有 `reasoning?: string`，流中也确实累积了 reasoning；但内部 `ApiMessage` 只声明 `role/content/tool_calls/tool_call_id/name`，构造下一轮 `apiMessages` 时没有填 `reasoning_content`。在思考模型先输出 reasoning 再决定调用工具的情况下，下一次请求可能要求将这段 reasoning 原样返回。MiniMax 的真实错误文本恰好给出了这一规则，因此 GPT 不能对工具协议给满分。修补方法很小：给 `ApiMessage` 增加 `reasoning_content?: string`，并在 `source.reasoning` 非空时赋值；同时以实测“现在几点”验证第二次请求。

第二项是可维护性。`ChatStorage` 用 JSON 整体存储 settings 和 conversations，`JSON.parse` 的异常被吞掉并回落默认值，能避免崩溃但会静默丢数据；应至少记录错误或做版本迁移。构建还报告 `preferences` 相关函数可能抛异常、`getContext`/`showToast` 已弃用，建议在下一次工程升级时按新 API 迁移。最后，应补一个纯函数测试集：给 SSE 切片输入、半个 UTF-8 字符、跨块 tool arguments、工具后的 reasoning 回传 body 和损坏历史 JSON。完成这些后，它可作为其余模型的基线实现。

## 八、跨模型技术问题分析

### 8.1 “stream: true” 与“真流式 UI”是两个不同层次

多数实现会在 JSON 里写 `stream: true`，但这只是服务端协商流式响应的请求参数。客户端是否流式，要看它是否用能在网络尚未结束时得到数据块的 HarmonyOS API，并将块一路传到状态管理与组件渲染。Qwen、MiMo 和 ERNIE 证明了反例：它们可以拿到格式为 SSE 的完整文本，再逐行调用解析函数；代码看似有“onContent”，但用户只会在服务端结束后看到结果。此类实现最多可以称“兼容 SSE 响应”，不能称“流式输出”。

一个正确最小路径应为：

```text
requestInStream
  -> dataReceive(ArrayBuffer)
  -> UTF-8 decoder + pending buffer
  -> 完整 SSE event
  -> JSON delta
  -> append reasoning/content
  -> 用新状态更新 ArkUI
  -> 用户在本轮结束前可看到文字
```

Kimi 表明只完成前五步也不够：它已有 requestInStream、buffer、解析和 accumulator，却把 accumulator 的文本留到请求结束才返回，漏掉了第六、七步。MiniMax 表明 UI 侧用“完成”提示代替内容渲染也不够。GPT、Claude、Gemini、DeepSeek、dsFlash、GLM、豆包在结构上都把网络增量回调接入了页面状态；Composer 则把完整结果交给显示队列模拟增量，不能算同一类真流式；其中 GPT 的首交稳定性最好，Gemini 的消息字段最完整。

### 8.2 SSE 边界、UTF-8 与工具分片

不要假定一次 `dataReceive` 对应一次 JSON。服务可能把一个 `data:` JSON 拆到两个 ArrayBuffer，也可能在一个 ArrayBuffer 放多个事件；换行可能是 `
` 或 `\r
`；中文 UTF-8 字节也可能被截断。最好的做法是 decoder 使用流式模式、保存未完成文本，按事件空行切边界，处理掉 `event:`/空行后再解析 data。GPT 用 `

` 做事件边界相对稳妥；豆包和 Kimi 用按行缓存也能工作，但需要仔细处理残留和 CRLF；仅对完整 `response.result` 或 `data.result` split 的 Qwen/MiMo/ERNIE 则没有“跨网络块”的问题，却已经失去实时性。

工具调用更复杂。服务端会先只给 `index`/id，再在之后的 delta 给函数名片段和 arguments 片段。只有在流结束或明确 finish reason 为 `tool_calls` 后才可以执行。按每个分片立即调用时间工具，会得到空函数名或只得到局部名称，正是 MiniMax 出现多次 `unknown tool:` 的典型原因。正确模型需要以 `index` 为 key，分别拼接 `id`、`function.name`、`function.arguments`；执行前校验 name 非空、arguments 可解析；执行后生成一条与 call id 对应的 tool message。

### 8.3 思考模式下的工具回合是本题最大的协议陷阱

“显示思考”并不只是将 `reasoning_content` 画到卡片。若模型在思考模式里发出工具调用，下一轮请求往往需要带回前一条 assistant message 的 reasoning 内容，以及该条消息的 `tool_calls`，然后再附上 `role: tool`、`tool_call_id` 的结果。结果记录中的 MiniMax HTTP 400 已给出服务端对此的明确要求。缺失该字段，第一轮普通聊天可能很好，问当前时间却会失败，这种问题极具迷惑性。

在最终代码中，Gemini、Composer 和 Kimi 的消息模型最接近正确：Gemini 直接在 `ChatMessage` 与 API message 中保留 `reasoning_content`，Composer 的 `ApiChatMessage`/`ChatManager` 也会在工具回合映射 `reasoning_content`，Kimi 同样有 `reasoningContent` 并映射到 `reasoning_content`。Claude、DeepSeek、dsFlash 和 GPT 都能保存或展示 reasoning，但构造下一轮 API assistant 消息时没有把 `reasoning_content` 作为正式字段带回；GLM、豆包、Qwen、MiniMax、MiMo也存在不同程度缺失，ERNIE 则根本没有完整工具消息模型。这个横向缺陷说明许多模型只理解了单轮流式响应，而没有把 API 看作严格的多回合协议状态机。

### 8.4 ArkTS 不是宽松 TypeScript

首轮错误数量高度集中在 ArkTS 的静态约束：未声明的对象字面量、对象字面量作为类型、不可推断数组、`ValueType` 直接当 string、错误导入的网络类型、错误的回调参数、可能为 null 的对象、误用 UI 组件和属性。这些不是偶然拼写错误；它们反映模型是否会先建立显式类/interface、是否会查阅 API 声明、是否理解 ArkUI DSL 与普通 JS 方法调用的区别。

高质量 ArkTS 实现应先写好 `ChatMessage`、`ToolCall`、`RequestBody`、`StreamChunk`、`Settings` 等声明，再让所有对象字面量对应这些声明；Preferences 的返回值要做 `String()` 或完整类型检查；可空会话要在一个 guard 后收窄；NetworkKit 回调要依据当前 SDK 的签名书写。GPT 的首次通过说明这种方法可行。GLM 最终修复也说明问题并不难，但首次交付前没有做过一次真实构建就是过程缺陷。

### 8.5 存储、API Key 和可发布性

全体方案基本都用 Preferences 保存 Key 与历史，满足题目“保存读取”与“用户自己填写”的最低要求。优点是实现简单、应用重启后数据仍在；缺点是 Key 常和会话 JSON 一样以明文落在普通 Preferences 中。生产版本至少应评估系统安全存储/加密能力、设置清除 Key 的入口、日志不打印 Authorization、历史数据长度上限和迁移版本。题目未要求这些，不把它们转换为致命扣分；但题目使用“完整 APP”“可落地”的语境，评测必须将其标注为上线前工作。

构建方面，十三个评分项目最终 HAP 打包都成功，但日志均指出没有 signingConfigs，签名步骤被跳过。若要真正在设备侧分发或上架，必须配置签名材料；本报告不把这一项目计为模型竞赛失败，因为示例仓库的默认签名状态相同，且题目核心是 DevEco 编译能力。

## 九、推荐的验收用例与回归门槛

后续若要将这组测试变为可重复的模型基准，建议统一使用下列验收表。每个用例都应记录构建输出、屏幕录制或截图、请求 body（脱敏 Key）以及响应状态；模型只有全部关键用例通过，才能被称为“完成”。

| 编号 | 用例 | 预期 | 关键失败信号 |
|---|---|---|---|
| T01 | 空 Key 点击发送 | 阻止请求并引导设置 | 静默无反应、发送空 Authorization |
| T02 | 保存 Key/模型/开关/深度后重启 | 值仍存在，关闭思考不显示/不发送深度 | 设置只停留在内存 |
| T03 | 开启思考发送较长问题 | 思考内容先后增量出现，回答在结束前逐步出现 | 完成后一次出现、只显示加载圈 |
| T04 | 关闭思考发送 | 无思考卡片，正文仍真流式 | 仍显示 reasoning 或请求字段错误 |
| T05 | 首轮后追问“再详细一点” | API 历史包含前一问答，回答能承接语境 | 重开为新对话、历史顺序倒置 |
| T06 | 退出/重启后打开历史 | 会话标题、消息、思考/工具展示一致 | 空白、乱码、只重载才刷新 |
| T07 | “现在几点”且开启思考 | 一次完整工具调用、时间结果、最终自然语言回答 | unknown tool、空工具、400、无限等待 |
| T08 | 连续两次时间工具调用 | 每轮 tool_call_id 一一对应且不污染后续 | 第二次 400、重复 tool result |
| T09 | 发送途中点击停止 | 请求释放，部分消息保留或明确取消，无卡死 | 后续无法发送、重复 onDone |
| T10 | 错误 Key/断网 | 用户看到可理解错误，恢复网络后可重试 | 只显示 `[object Object]`、状态永远 loading |
| T11 | 长中文、超长工具结果、大字体 | 无乱码、无文字越界、可滚动 | 背景文字、屏幕外文本、重叠 |
| T12 | DevEco clean build | 无 ArkTS Compiler Error，HAP 产物存在 | 仅 IDE 临时缓存可过、命令行失败 |

其中 T07 是本题的判别力最高用例。它同时验证模型设置、SSE、reasoning、工具增量合并、本地时间、消息序列化、第二次请求、最终 UI 和历史保存。当前多份方案只通过了“工具定义存在”或“第一轮收到了 tool_calls”，却没有通过 T07。将它放在开发早期，可以避免在布局和页面上投入大量工作后才发现协议根基错误。

建议为这些用例建立本地 mock SSE 服务或纯函数夹具。虽然 ArkTS 本身不使用 OpenAI SDK，但 SSE parser、消息序列化和工具回合状态机都可以脱离真实网络测试：输入人为拆分的数据块，断言页面回调顺序和下一轮 body。真实 DeepSeek Key 仅用于少量集成验收，避免用网络波动掩盖实现错误。

## 十、对各模型的后续处理建议

### 10.1 可直接作为基线：GPT-5.5

保留架构，优先补 reasoning_content 回传与工具回归；之后处理 ArkTS 警告、存储错误可观测性、签名和专项测试。它不需要重写 UI 或传输层。若要继续演进，建议将页面内嵌的设置/历史 Sheet 拆成独立组件，并将会话状态抽成可测试的 store，但这属于优化，不是当前交付的前置条件。

### 10.2 可作为第二基线或局部借鉴：Claude、Gemini、DeepSeek、GLM、dsFlash

Claude 可以作为“工具消息顺序修正确后可用”的第二基线，优先补 `reasoning_content` 回传和生产态错误脱敏；Gemini 可保留最终的消息模型和工具回合，重点补 UTF-8 流式解码、SSE event 边界和多工具边界测试；DeepSeek 可保留最终的流式、工具和存储结构，但要把 INTERNET 权限、请求状态机和 reasoning 回传补成首交级别的稳定能力；GLM 可保留 SSEParser、数据模型和页面，先补消息字段和工具回合测试；dsFlash 可保留 `requestInStream` 和工具循环，但要先补 `reasoning_content`、SSE event 边界、最大工具轮数和 null delta 过滤。五者都不应再重写主框架，应围绕 T03 真流式和 T07 时间工具做小步回归。

### 10.3 可保留局部设计但不能直接验收：Composer、豆包、Kimi

Composer 可保留 `ChatManager` 的工具消息顺序、`reasoning_content` 回传和工具轮次保护，但必须把普通 `request()` 加伪流式队列替换为真正 `requestInStream`；豆包可保留请求与存储分层，但应先修 UI 布局，避免继续在不可读界面上增加功能；Kimi 可保留其最正确的 reasoning/tool 消息设计，重点把 delta 回调接到 UI，并收紧会话空值状态。三者都有值得复用的协议片段，但当前运行体验不足以作为完整交付。

### 10.4 需要替换核心传输/状态链：Qwen、MiniMax、MiMo、ERNIE

Qwen、MiMo 和 ERNIE 首先替换普通 `request()` 路线，不应试图在完整 body 解析器上添加“逐字动画”来伪装流式；MiniMax 首先统一 assistant reasoning、tool calls、tool result 三种消息的单一事实来源，删除“完成统计代表用户已看到内容”的逻辑。ERNIE 还必须先恢复一个可见主界面，并停止通过反复推翻同一文件来处理局部报错。四者都应该用显式消息类和 `JSON.stringify`，减少字符串拼接 JSON 的范围。完成最小链路后才讨论界面美化和多会话功能。

### 10.5 对评测流程本身的建议

未来如果继续测试模型开发能力，建议在题目旁提供：目标 HarmonyOS API 版本、允许查阅的离线 SDK 文档、初始 `local.properties` 示例、一次 clean-build 命令、以及不含答案的最小 `requestInStream` 类型签名定位提示。这样测试重点会更集中在工程推理与实现，而不是让模型因为环境变量或不透明 SDK 版本浪费全部时间。与此同时，应明确禁止读取相邻模型目录，避免模型互相污染结果；允许查 SDK 文档，但应记录查询范围。

评分时仍应保留本报告的“首交编译错误无上限扣分”规则，因为它极好地区分了会自检的实现和先堆量后修补的实现。还应把 `git diff` 统计、T07 工具回合录屏和最终 clean build 作为强制产物。仅凭模型的口头“已经实现”或一个 `BUILD SUCCESSFUL` 截图，无法评价完整 APP；文心这次的负分尤其说明，修复时引入的新编译错误和编辑工具误用必须进入过程评分。

## 十一、最终结论

这组结果清楚展示了生成式模型在完整软件任务上的主要分化点。多数模型能够迅速铺出页面、设置、模型名、历史和“工具调用”的表面结构；真正拉开距离的是四件事：是否尊重 ArkTS 的严格类型与真实 SDK 签名，是否区分服务端 `stream: true` 和客户端真流式，是否把 reasoning 当成多回合协议数据而不是一次性 UI 文本，以及是否在每次修复后用用户可见的端到端流程验证。

GPT-5.5 的表现最接近可交付工程：首次构建、结构、交互和运行质量均领先；它应被选作后续迭代基线，但在正式上线前也必须修工具 reasoning 重传并补回归。Claude 是新增结果里最稳的真流式方案之一，工具协议最终修通，但过程里有猜根因和 reasoning 回传风险；Gemini 的最终协议模型很亮眼，能把 `reasoning_content`、`tool_calls` 和工具结果串入下一轮，但首交错误和一次显示状态断裂使它低于 Claude；Composer 编译扣分很低，且工具协议模型不错，但伪流式使它停在中上游；DeepSeek 最终跑通，基础完成度高，但首交错误和权限/无响应链路暴露自检不足。GLM 是“首交失误较多但修复收敛好”的第二梯队；Qwen 有较全产品壳但核心流式误选 API；豆包和 Kimi 各有可复用的技术亮点，却分别被 UI 可用性和状态链断裂拖住；dsFlash 最终业务跑通但全程 50 个 ArkTS 编译错误和两次运行级修复暴露首交自检不足；MiniMax 与 MiMo 的问题集中体现为把内部代码路径当作真实产品完成；ERNIE 则是最终可编译但主界面纯白、全程 102 个编译错误的反面样本。

因此，本题最重要的评分结论不是“所有最终项目都能构建”，而是：**只有一个模型在首次可构建、核心流程、工程结构和运行体验四项上同时表现出稳定完成度；Claude、Gemini、Composer 和 DeepSeek 可进入第二基线或局部基线打磨，其余方案需要依据明确的协议、传输、UI 或过程问题继续修复，不能仅凭最终 BUILD SUCCESSFUL 作为已完成任务验收。**
